push_swap
como duas pilhas aprendem a se ordenar

Um programa em C que recebe uma lista de números desordenados e a organiza usando apenas duas pilhas (A e B) e um conjunto restrito de movimentos — trocar, rodar, empurrar. Este documento explica, em camadas visuais, como cada arquivo do projeto contribui para essa tarefa.

01

O que este programa faz, em linguagem simples

Sem termos técnicos: imagine duas pilhas de cartas na mesa.

Você entrega ao programa uma lista de números embaralhados, por exemplo 4 67 3 87 23. Esses números viram uma pilha de cartas (Pilha A). O programa só pode mexer nas cartas de quatro formas: trocar as duas do topo, girar a pilha para cima ou para baixo, e empurrar a carta do topo para a outra pilha (Pilha B) ou trazê-la de volta. Usando só esses movimentos — na ordem certa — ele reorganiza os números do menor para o maior na Pilha A, e imprime cada movimento feito, um por linha, no terminal.

A parte inteligente do projeto é escolher a estratégia certa conforme o tamanho da lista: poucos números pedem uma solução direta; muitos números pedem um método mais elaborado para não desperdiçar movimentos.

02

Mapa dos arquivos do projeto

Cada arquivo .c tem uma responsabilidade única. push_swap.h é o "contrato": declara todas as funções para que os arquivos possam se enxergar.

main.c
Ponto de entrada. Lê os argumentos, decide se já está ordenado e escolhe a estratégia de ordenação pelo tamanho.
parser.c
Converte texto em números (com validação de overflow) e detecta duplicados.
stack_utils.c
Cria nós, monta a lista circular, libera a memória no final.
indexer.c
Atribui a cada valor um "índice" — sua posição relativa se a pilha estivesse ordenada (0 = menor, N-1 = maior).
disorder_metric.c
Calcula quantas "inversões" (pares fora de ordem) existem na pilha. Contém também get_stack_size, usada por todo o projeto.
math_utils.c
Uma raiz quadrada inteira própria, usada para dimensionar os "blocos" dos algoritmos médio/complexo.
op_swap_push.c
Implementa de fato o "trocar" (swap) e o "empurrar" (push) entre pilhas, mexendo nos ponteiros da lista.
op_rotate.c
Implementa "girar" a pilha para cima (rotate) e para baixo (reverse rotate).
op_wrappers_a.c / b.c
Camada fina que chama as operações acima e imprime o nome do comando executado (sa, ra, pb...).
simple_sort.c
Ordena 3 elementos diretamente e listas pequenas (4–10) por seleção de mínimo.
medium_sort.c
Estratégia por "blocos" (chunks) para listas de tamanho médio (11–100).
complex_sort.c
Mesma ideia de blocos, ajustada para listas grandes (101+).
03

Diagrama de arquitetura — quem depende de quem

As setas indicam "chama funções de". push_swap.h (faixa lateral) é o contrato incluído por todos os arquivos.

Observação de arquitetura — não existe um wrapper sb (swap na pilha B). A função genérica swap() existe, mas só é exposta via sa. Isso é coerente com os algoritmos aqui: eles nunca precisam trocar os dois primeiros elementos de B, só de A.
Observação de arquitetura — calculate_disorder_metric (em disorder_metric.c) é declarada e implementada, mas nenhum outro arquivo a chama. É uma função "órfã" — provavelmente preparada para uso futuro em depuração ou em uma heurística de escolha de algoritmo.
04

A estrutura de dados: uma pilha circular

Cada pilha não é um array — é uma lista duplamente ligada e circular. O último elemento aponta de volta para o primeiro, o que torna "girar" a pilha uma operação muito barata: só se move o ponteiro de cabeça.

CampoTipoPapel
valueintO número original informado pelo usuário.
indexintPosição que o valor teria numa pilha ordenada (calculado por set_stack_indices). É o que os algoritmos usam para decidir movimentos, sem comparar valores brutos toda hora.
next / prevponteirosLigam o nó aos vizinhos, formando o anel. Permitem girar a pilha apenas movendo o ponteiro de cabeça — sem copiar dados.

09

Resumo: como as funções conversam entre si

Tudo começa em main.c, que delega a leitura dos argumentos para parser.c (que valida números e detecta duplicados) e a montagem da pilha para stack_utils.c. Antes de ordenar, is_already_sorted evita trabalho desnecessário quando a lista já está em ordem.

A decisão de qual arquivo de ordenação usar é tomada apenas por select_sort_strategy, olhando o tamanho da pilha. Os três algoritmos maiores (sort_small, medium_sort, complex_sort) dependem de indexer.c para saber a posição relativa de cada valor e de disorder_metric.c apenas para medir o tamanho da pilha (get_stack_size) — os dois algoritmos de blocos também usam math_utils.c para calcular o tamanho ideal do bloco via raiz quadrada.

Nenhum algoritmo de ordenação mexe diretamente nos ponteiros da lista. Todos passam pela camada de op_wrappers_a.c e op_wrappers_b.c, que são finas: chamam a operação real (em op_swap_push.c ou op_rotate.c) e imprimem o nome do comando. Essa separação é o que permite ao programa ao mesmo tempo ordenar a pilha e produzir, como efeito colateral, a lista de comandos exigida pelo enunciado do projeto.

Por fim, main.c libera toda a memória das duas pilhas com free_stack, garantindo que nenhum nó fique órfão, e encerra.
